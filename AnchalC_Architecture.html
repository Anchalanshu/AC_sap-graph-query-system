01 / Overview

System at a Glance

The core design principle: AI is an interpreter, not a data source. Every answer the system gives must be grounded in real data from the database. The LLM's job is to translate human intent into precise SQL, then translate raw results back into human language. It never invents facts.

Key design decision:

SQLite was chosen over a native graph database (Neo4j/ArangoDB) because the dataset's relationships are relational at heart — a join between sales_orders and sales_order_items is faster and more queryable than a graph traversal for this scale. NetworkX is used purely as a visualization layer — a projection of the relational data, not the query engine.

02 / AI Workflow

How AI Enters the Workflow

The AI is invoked at exactly three points in the request lifecycle. This surgical integration means the system degrades gracefully if the LLM is slow or unavailable — the database and graph still work independently.

01

User submits natural language query INPUT

The frontend captures the user's question (e.g. "Which plant produced the most items last month?") along with the last 4 messages of conversation history for context. Sent via POST /api/query.

02

Guardrail pre-check GATE

Before the LLM is invoked, a keyword-based filter checks if the query is domain-relevant. This runs in <1ms and prevents wasting API quota on off-topic requests. If blocked, response is returned immediately without touching the LLM.

03

AI invocation #1 — SQL Generation LLM CALL

The query + schema + conversation history are sent to Gemini. The LLM responds with a structured JSON containing a SQLite-compatible SQL query. This is the most critical AI integration point — prompt engineering here determines correctness.

04

SQL validation and execution DATABASE

The generated SQL is validated (SELECT-only, no DDL/DML allowed) before execution. Results are fetched from SQLite and serialized. If SQL is invalid, fallback logic triggers without a second LLM call.

05

AI invocation #2 — Answer Synthesis LLM CALL

The raw SQL result rows are sent back to Gemini with a formatting prompt. The model synthesizes a concise, human-readable answer. It also identifies entity IDs mentioned in the answer for graph highlighting.

06

Graph highlight + response streaming OUTPUT

The answer streams back to the frontend word-by-word via Gemini's streaming API. Simultaneously, node IDs extracted from the answer are sent to the graph — matching nodes pulse yellow, and the camera zooms to frame them.

Why two separate LLM calls instead of one?

Separating SQL generation from answer synthesis gives two major benefits: (1) you can validate and retry the SQL independently without regenerating the whole answer, and (2) the answer synthesis prompt can be given the actual data, not just a description of it — making the response more accurate and grounded.

03 / Prompt Design

Prompt Engineering Strategy

Prompt design is where the system's intelligence lives. A poorly designed prompt produces hallucinated SQL, ignored guardrails, and vague answers. The prompts below are engineered with five principles: Role clarity, Schema grounding, Strict output format, Explicit rules, and Context injection.

Prompt #1 — SQL Generation Prompt

This is the most critical prompt. It must make the model output only valid SQLite syntax, nothing else.

SYSTEM — SQL Generation Prompt · sent on every query
Role Definition
You are a SQL expert for a SAP ERP system. You translate natural language questions into valid SQLite queries. You are NOT a general assistant — you only answer questions about the provided database.
Schema Injection (dynamic)
Database schema:
TABLE sales_orders: sales_order TEXT PK, transaction_currency TEXT, total_net_amount REAL
TABLE sales_order_items: id INT PK, sales_order TEXT FK, sales_order_item TEXT,
  item_category TEXT, material TEXT, requested_quantity REAL,
  quantity_unit TEXT, net_amount REAL, material_group TEXT,
  production_plant TEXT, storage_location TEXT,
  rejection_reason TEXT, billing_block_reason TEXT
TABLE materials: material TEXT PK, material_group TEXT
TABLE plants: plant_id TEXT PK
Hard Rules
RULE 1: Only generate SELECT statements. Never INSERT, UPDATE, DELETE, DROP.
RULE 2: If the question is NOT about this SAP dataset, set sql to null.
RULE 3: Always return valid JSON. No markdown, no explanation outside JSON.
RULE 4: Use LIMIT clauses for queries that could return many rows.
RULE 5: Empty strings mean no rejection/block — filter with != '' for active blocks.
Output Format (enforced)
Return ONLY this JSON:
{
  "sql": "SELECT ... FROM ... WHERE ...",
  "explanation": "Plain English description of what this query does",
  "is_domain_relevant": true
}
Conversation History (injected dynamically)
Previous context:
[User]: Which plant has the most items?
[Assistant]: Plant 1920 has 45 items...

Current question: {{ user_query }}
Why inject the schema every time?

LLMs don't have persistent memory between calls. Without the schema, the model would guess column names and hallucinate table structures. Schema injection is non-negotiable — it's what makes the SQL generation reliable. The schema is <300 tokens, well within budget.

Prompt #2 — Answer Synthesis Prompt

After SQL executes, this prompt converts raw rows into a natural language response. It's intentionally simple — the model's only job here is formatting, not reasoning.

SYSTEM — Answer Synthesis Prompt · sent after SQL execution
Task
Convert the following SQL query results into a clear, concise answer to the user's question. Be specific — use the actual numbers and IDs from the data. Do not add information not present in the results.
Injected Data
User question: {{ user_query }}
SQL executed: {{ sql }}
Results ({{ row_count }} rows):
{{ results_json }}
Output Requirements
1. Answer in 2-4 sentences max.
2. Mention specific IDs/values from the data.
3. End with: "NODES: [id1, id2, ...]" listing any sales order or material IDs mentioned.
4. If results are empty, say "No records found matching your query."
// User asked: "Which material has the highest total order value?" // LLM Answer Synthesis output: "Material S8907367001003 has the highest total order value at ₹1,24,832.50 across 12 sales order items. It belongs to material group ZFG1001 and is primarily produced at plant 1920." // Node highlighting extraction: NODES: ["S8907367001003", "ZFG1001", "1920"]

Prompt Design Principles Summary

🎯

Role Clarity

Each prompt starts by defining exactly what the model IS and IS NOT. "You are a SQL expert, not a general assistant" prevents the model from answering off-topic questions even if the guardrail is bypassed.

📋

Schema Grounding

The full database schema is injected on every call. Without this, column names are guessed. With it, the model generates SQL that actually executes.

📦

Structured Output

Requiring JSON output with exact field names makes parsing deterministic. No regex extraction needed — just JSON.parse().

🔒

Explicit Prohibitions

Rules like "Never INSERT, UPDATE, DELETE" are stated directly. LLMs respect explicit prohibitions better than implicit expectations.

🧠

Context Injection

The last 4 message pairs are injected as conversation history. This enables follow-up questions like "tell me more about that order" to resolve correctly.

⚡

Token Efficiency

Prompts are kept minimal. The schema is ~250 tokens. History is capped at 4 turns (~800 tokens). This keeps latency low and stays within free tier limits.

04 / System Integration

Component Integration Strategy

Each component is loosely coupled via well-defined interfaces. The frontend doesn't know how the graph was built. The LLM doesn't know how its SQL will be executed. This separation means each layer can be swapped independently.

Frontend → Backend

  • POST /api/query — natural language + history
  • GET /api/graph — graph nodes and links JSON
  • GET /api/schema — table schema for debugging
  • GET /health — liveness check
  • CORS configured for localhost:5173
⟷

Backend → Database

  • SQLite via Python's built-in sqlite3 module
  • Connection pooled via FastAPI lifespan
  • Read-only connection mode for query execution
  • Schema loaded once at startup, cached in memory
  • Parameterized queries prevent SQL injection

Backend → Gemini API

  • google-generativeai Python SDK
  • Model: gemini-1.5-flash (fast, free tier)
  • Async calls — won't block the event loop
  • Streaming enabled for answer synthesis
  • 10s timeout with exponential backoff retry
⟷

Backend → Graph Builder

  • NetworkX graph built once on server startup
  • Cached in memory — rebuilt on /api/graph refresh
  • Nodes sampled to max 150 for UI performance
  • Node highlight IDs from LLM matched against graph
  • Graph serialized to JSON (nodes + links arrays)

05 / Fallback Mechanisms

What Happens When AI Fails

The system is designed so that an LLM failure is a degraded experience, not a broken one. There are four distinct failure modes, each with a specific recovery strategy.

Failure Mode Detection Fallback Action User Experience
SQL Invalid
LLM returns malformed SQL		 SQL parse check before execution; detect non-SELECT statements Return a pre-written error message. Log the bad SQL for analysis. Don't attempt execution. "I couldn't generate a valid query for that. Try rephrasing." — graph unchanged.
Empty Results
Valid SQL, zero rows		 len(results) == 0 check after execution Skip answer synthesis LLM call. Return template: "No records found." Instant response, no wasted API call. Saves latency.
API Timeout
Gemini takes >10s		 asyncio.timeout(10) wrapper around every API call Retry once with 2s backoff. On second failure, return fallback string without LLM. "The AI service is slow right now. Try again in a moment." — graph and data still work.
Bad JSON Response
LLM returns non-JSON		 try/except json.JSONDecodeError on every parse Attempt regex extraction of SQL from response text. If that fails, log and return error. "Couldn't parse the AI response. Please try rephrasing your question."
Off-topic Query
Guardrail triggered		 Level 1: keyword check (<1ms). Level 2: LLM self-classification Immediate return at Level 1. For Level 2, model returns is_domain_relevant: false. Styled guardrail response with orange border: "This system only answers questions about the SAP ERP dataset."
async def safe_query(question: str, history: list) → dict: # Layer 1: instant keyword guardrail if not is_domain_relevant(question): return { "answer": GUARDRAIL_MSG, "is_domain_relevant": False } # Layer 2: LLM SQL generation with timeout try: async with asyncio.timeout(10): llm_response = await generate_sql(question, history) except TimeoutError: return { "answer": TIMEOUT_MSG, "error": "timeout" } # Layer 3: SQL validation before execution sql = llm_response.get("sql") if not sql or not is_safe_select(sql): return { "answer": INVALID_SQL_MSG, "sql": sql } # Layer 4: execute and handle empty results results = execute_query(sql) if not results: return { "answer": "No records found matching your query.", "results": [] } # Happy path: synthesize answer answer = await synthesize_answer(question, sql, results) return { "answer": answer, "sql": sql, "results": results }
SQL Injection Prevention:

The system never executes LLM-generated SQL directly. Every query goes through is_safe_select() which validates: (1) starts with SELECT, (2) contains no semicolons after the first statement, (3) contains no DDL keywords — DROP, CREATE, ALTER, INSERT, UPDATE, DELETE. This makes it safe even if the LLM is adversarially prompted.

06 / Reliability Engineering

Building for Production Reliability

A system that works in demos but fails under real use is not production-grade. These reliability patterns are baked into the architecture from day one.

🔁

Idempotent Graph Builds

The graph is built from SQLite, not from the JSONL file directly. Running the ingest script twice won't create duplicate nodes — SQLite's PRIMARY KEY constraints and INSERT OR REPLACE semantics ensure clean rebuilds.

⚡

Async Everything

FastAPI's async endpoints mean the Gemini API call (800ms-2s) never blocks other requests. Multiple users can query simultaneously without queueing. The event loop stays free during I/O waits.

💾

Schema Cached at Startup

The database schema is read once on server startup and stored in memory. Every LLM call injects this cached string — no per-request DB metadata queries.

🔒

Read-Only Query Execution

The SQLite connection used for LLM-generated queries is opened in read-only mode. Even if the guardrails fail and a destructive query is generated, it cannot execute.

📊

Graph Sampling

The graph visualization is capped at 150 nodes. For larger datasets, nodes are sampled by entity type to maintain proportional representation. This keeps the React rendering performant.

🗂️

Conversation Memory Cap

History is capped at 4 message pairs (~800 tokens). This prevents context window overflow while still enabling meaningful follow-up questions. Older context is dropped from the front.

Guardrail Layer Design

Two independent guardrail layers means bypassing one still hits the other. Defense in depth.

L1
Keyword Filter
Runs in <1ms. Checks if any domain keyword appears in the query. Zero LLM cost. Blocks obvious off-topic queries like "write me a poem" or "what is the capital of France".
Fast
L2
LLM Self-Classification
The system prompt instructs Gemini to return is_domain_relevant: false for anything not about the SAP dataset. Catches clever rephrasing that passes keyword filters (e.g. "How many stars are in the sky?" doesn't contain ERP keywords but is clearly off-topic).
Smart
L3
SQL Safety Validator
Even if both L1 and L2 pass, the SQL is validated before execution. Only SELECT statements allowed. This is the last line of defense against prompt injection attacks where a user tries to embed SQL commands in their query.
Critical
07 / Architecture Layers

The system is organized into five clean layers. Each layer has a single responsibility and communicates with adjacent layers only through defined interfaces. This makes the system testable, extensible, and maintainable.

What makes this architecture stand out for a Forward Deployed role:

In enterprise ERP deployments, systems must degrade gracefully (a plant can't stop because an API is down), must handle adversarial inputs (users will try to break it), and must be explainable to both technical and non-technical stakeholders. This architecture addresses all three: layered fallbacks for graceful degradation, three-layer guardrails for adversarial resilience, and the SQL-explanation feature for transparency.
